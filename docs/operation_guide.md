## 2. Day1–Day7 每天要做什麼

### 每天共通（固定節奏，約 30–45 分鐘人力）

- [ ] **19:30** 確認當日劇情在站上正常解鎖（若採每日解鎖機制）
- [ ] **20:00** 發當日主貼文（劇情鉤子＋連結）— 依第 3 節流程
- [ ] **21:00–23:00** 值班：回覆留言（第 5 節）、監看異常
- [ ] **23:30** 數據快照（第 4 節）＋ 記錄當日一行日誌（事故／玩家反應亮點）

### 每日重點差異

- [ ] **Day1（06-11 四）**：上線日。發「收到情書的第一個晚上，我家門口站了一個偵探。」鉤子文；緊盯前 3 小時 console 錯誤回報；置頂「怎麼玩」說明。
- [ ] **Day2（06-12 五）**：失蹤成真日。轉發玩家「預告成真了！」反應；確認 c02/c03/c04 調查選項取得率正常。
- [ ] **Day3（06-13 六）**：週末＋金句日（「我不是第一次失去你」）。加發一張金句卡圖；週末流量高，留言回覆配額加倍。
- [ ] **Day4（06-14 日）**：偵探成為嫌疑人。社群會吵「偵探是不是犯人」——**官方不暗示答案**，只丟開放式問題（「你按下了相信，還是懷疑？」）。
- [ ] **Day5（06-15 一）**：核心抉擇日（trust/suspicion 影響最大）。發投票貼文「相信／懷疑」；這天的投票數據是 Day8 付費轉換的素材。
- [ ] **Day6（06-16 二）**：L7 反轉日。嚴防劇透：留言區劇透一律折疊／回覆「請用防雷格式」；發文只用情緒不用情節（「今天的更新，建議準備好再點開」）。
- [ ] **Day7（06-17 三）**：結局日。發「你走到了哪個結局？」分享活動（玩家貼結局分享圖）；當晚預告「明天有給想知道全部真相的人的東西」——為 Day8 鋪路，**話術只說補完，不說解鎖真相**。

## 3. 每天 SNS 發文流程

- [ ] 1. 開 `social_calendar.csv`，找到今日 `date_jst` 的列
- [ ] 2. 核對三件事：文案無劇透超標、連結帶正確 UTM（`s1_dayN`）、配圖對應 `assets/`
- [ ] 3. 安全自查 10 秒：無未成年暗示、無勒索話術、無「真人感」誤導（簡介需保持「虛構互動故事」標示）
- [ ] 4. 取得發佈批准（未批准 → 維持 draft，當日改為僅站內更新）
- [ ] 5. 20:00 JST 發主貼文（X 為主；IG/Threads 同步可延後 10 分鐘）
- [ ] 6. 發後 15 分鐘內自我回覆一則補充（玩法連結或前情提要），墊高曝光
- [ ] 7. 在 CSV 把該列 `status` 改為 `posted` 並填實際發文網址

## 4. 每天數據檢查流程（23:30 JST）

零後端 → 數據來源有限，固定看這五項並記入 `tmp/ops_log.md`（或任一表格）：

- [ ] 1. **SNS 端**：當日主貼文 曝光／互動數／連結點擊（平台內建分析）
- [ ] 2. **站台端**：GitHub Pages 無內建分析；看 repo Insights→Traffic 的 Views/Unique（Cloudflare Pages 則看 Web Analytics，免費且無 cookie，符合不收個資原則）
- [ ] 3. **留言端**：手動數三類留言數 — 劇情討論／技術問題／負面不適
- [ ] 4. **轉化端**（Day8 後）：Stripe Test/正式 dashboard 的 Payment Link 點擊與完成數
- [ ] 5. **判斷**：對照基準 — 日活下滑 >40% 連續兩天 → 啟動第 9 節觸及改善；技術問題留言 ≥3 則同症狀 → 啟動第 7 節 bug 流程

## 5. 玩家留言回覆原則

- [ ] **回什麼**：玩法疑問（24h 內回）、bug 回報（當日回）、感想（點讚＋短回，優先回有截圖的）
- [ ] **不回什麼**：劇透追問（回「偵探不劇透 🌧」固定句）、攻略對錯爭論（不當裁判）、釣魚／引戰（不回，必要時隱藏）
- [ ] **身分原則**：以「官方帳號／製作組」口吻回覆，**永不以角色第一人稱私訊玩家**（AI 不假裝真人；角色口吻僅限公開的演出型貼文且明顯標示為作品內容）
- [ ] **不索取**：永不要求玩家提供真名、地址、電話、照片等個資；玩家主動貼出 → 提醒刪除
- [ ] 固定回覆範本備三句：玩法說明連結／bug 回報格式（瀏覽器＋截圖＋第幾天）／感謝語

## 6. 付費抱怨處理

- [ ] **「付了沒解鎖」**：先道歉→請對方提供付款完成頁截圖（遮卡號）→人工發送解鎖碼→24h 內結案
- [ ] **「覺得不值」**：不爭論價值，主動退款（Stripe 一鍵 refund），並記錄原因——MVP 期退款是便宜的市調
- [ ] **「為什麼要錢」**：固定話術——「Day1–7 主線與兩個結局完全免費，付費內容是給想看完整真相補完的玩家的支持選項」；**禁止**任何「不買就看不懂」的暗示
- [ ] **「解鎖碼被分享了」**：MVP 已知取捨，不公開追究玩家；量大時換碼（見 deployment_guide.md §4）
- [ ] 同類抱怨 ≥3 件 → 升級為產品問題，寫入 Day10 復盤清單

## 7. 故事 bug 修法

- [ ] 1. **分級**：A 卡死（無法進結局／JSON 解析失敗）— 2h 內修；B 判定錯（數值/線索條件不符 brief §8.3）— 當日修；C 錯字／顯示 — 併入隔日更新
- [ ] 2. **重現**：依玩家回報的選項序列在本機重走；對照 `brief/season1.md` 確認「正確行為」（brief 是唯一真相來源，**不得為了修 bug 擅改結局閾值**）
- [ ] 3. **修復**：只改 `content/story.json` 或 `public/app.js`；改前備份原檔
- [ ] 4. **驗證**：本機重跑受影響路徑＋normal_end 一條（迴歸）；`python -m json.tool` 過
- [ ] 5. **部署**：push → 1–3 分鐘生效；必要時 story.json 加 `?v=` 參數破快取
- [ ] 6. **公告**：影響存檔或結局的 bug，發一則更新說明＋向受影響玩家道歉；記入 ops_log

## 8. 玩家說內容讓他不舒服

- [ ] 1. **先承接不辯護**：「謝謝你告訴我們，抱歉讓你有不好的體驗」——不解釋「這只是劇情需要」
- [ ] 2. **判斷類型**：
  - 驚悚/情緒強度問題（如記憶刪除、病逝題材觸發個人經驗）→ 告知內容分級說明位置；評估是否在 `index.html` 加強內容提示（「含懸疑、微驚悚、親人離世題材」）
  - 涉及紅線疑慮（被解讀為自傷美化等）→ 視為 A 級事件：當日內帶著 `prompts/10_safety_reviewer.md` 標準重審該段文本，逾線即改寫熱修
- [ ] 3. **絕不**：嘲諷玩家「太敏感」、在公開區與玩家辯論
- [ ] 4. 玩家若透露自身處於危機狀態 → 以製作組身分溫和回應並附上在地求助資源（台灣安心專線 1925 等），**不扮演陪伴或治療角色**
- [ ] 5. 全部記入 ops_log，Day10 復盤必審

## 9. SNS 觸及不好怎麼改

連續兩天主貼文曝光低於 Day1 的 50% 即啟動，按順序試（一次只改一項，隔日看數據）：

- [ ] 1. **換鉤子型態**：純文字金句 → 對話截圖卡（Day N 的 SNS 截圖台詞本來就是為此設計，例如 Day2「照片裡我在笑」）
- [ ] 2. **換時段**：20:00 → 12:30（午休）或 22:30（睡前），A/B 兩天
- [ ] 3. **改互動型**：宣告式貼文 → 提問／投票式（「你會相信不肯辯解的偵探嗎？」）
- [ ] 4. **借力玩家**：轉發玩家結局分享圖＋感想（UGC 觸及通常優於官方文）
- [ ] 5. **平台再分配**：X 不動就把同素材改直式圖發 IG/Threads 試水
- [ ] 6. 三輪後仍無起色 → 接受本季自然量，把學習寫進 Day10 復盤，**不買廣告**（不花真錢）

## 10. Day8（06-18）如何推 True Case File

- [ ] 1. **時機**：Day7 結局上線滿 24h 後（玩家已有結局、知道自己「沒拼出全貌」的缺口感最強）
- [ ] 2. **主打對象**：走到 normal_end 的玩家（最大群體＋最強補完動機）
- [ ] 3. **貼文話術（缺口型，非勒索型）**：
  - ✅ 「千夏為什麼選了『信』？同意書折角的那行小字是什麼時候寫上的？True Case File：完整證據鏈＋千夏視角時間線＋偵探三年間的委託紀錄。NT$120 / ¥500」
  - ❌ 禁用：「不買就永遠不知道真相」「偵探在等你救他」
- [ ] 4. **內容承諾要明確**：列出檔案實際包含項（案件檔案／日記／Secret End 入口），與 `content/monetization.md` 一致
- [ ] 5. **免費玩家保障同貼重申**：「主線與 Normal/Bad End 永久免費」
- [ ] 6. 站內 `ending.html` 的 normal_end 結尾頁同步放入口（一個按鈕即可，不彈窗轟炸）
- [ ] 7. 全程 Stripe 仍為 Test mode，除非已獲明確批准上真商品

## 11. Day10（06-20）如何做復盤

- [ ] 1. 收齊數據：7+3 天的 SNS 曝光/點擊、站台流量、留言三分類統計、五結局到達分佈（站內若有埋分享圖，可由分享圖比例粗估）、付費點擊/完成、退款數
- [ ] 2. 按 `prompts/13_analytics_reviewer.md` 產出復盤報告，固定回答五題：
  - 玩家從哪天開始流失？（D1→D7 留存曲線）
  - 哪一天的 cliffhanger 帶來最高隔日回流？
  - normal_end vs bad_end 比例是否健康（bad_end 過高＝線索引導不足）
  - 付費轉換卡在「看到→點擊」還是「點擊→完成」？
  - 三件做對的事／三件下季必改的事
- [ ] 3. 產出物：`content/retro_s1.md`＋一頁摘要；把「必改」直接寫成 Season 2 的需求項
- [ ] 4. 對外發一則感謝文（含玩家數據彩蛋，如「68% 的人選擇了相信」）

## 12. 要不要做第二季的決策標準

Day10 復盤時逐項打分，**A 類達 2 項以上 → 做；A 類 0–1 項但 B 類全達 → 小成本再試一季；其餘 → 停，轉做別的題材實驗**：

- [ ] **A1 留存**：Day1 開玩者中 ≥30% 玩到 Day7 結局
- [ ] **A2 自傳播**：出現未經官方邀請的玩家自發貼文/分享圖 ≥20 則
- [ ] **A3 付費意願**：True Case File 點擊→完成轉換 ≥5%（或測試模式下明確的「想買」留言 ≥10 則）
- [ ] **B1 成本**：本季每日營運人力實際 ≤1 小時（系統可持續）
- [ ] **B2 管線**：用 content_pipeline.md 產第二季大綱實測 ≤2 天可出可玩初稿
- [ ] **B3 安全**：本季零紅線事故（無安全類 A 級事件）
- [ ] 決策本身寫入 `tmp/to_claude.md` 與復盤報告，附依據數字
