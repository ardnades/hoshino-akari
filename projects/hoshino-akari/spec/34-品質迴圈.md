# 34 — 品質迴圈（自動 QA・少打擾）

把「盡量推高遊戲品質」做成可重複的迴圈：自己挑項目、自己做、自己跑 QA，
只有在**真的需要你的眼睛或你的決策**時才打擾你。其餘例行打磨只記進度日誌。

## 一個迴圈循環

```
PICK  → 從「自動佇列」取最高 ROI 的一項（跳過「需你的眼睛」佇列）
BUILD → 最小改動實裝
QA    → 依序跑關卡（見下），任何一關紅 → 修或回退
GATE  → 通過且客觀可確認 → 記日誌，繼續下一項（不打擾）
        通過但觸及人眼/決策 → 停，報告
STOP  → 碰到停止條件 → 報告
```

## QA 關卡（每輪，便宜→貴）

1. `node --check` 改動到的 JS。
2. `node projects/hoshino-akari/tools/validate_story.js` → 0 error；warning 須解釋或為既有。
3. runtime：preview 載入、console 乾淨、對改動畫面 `preview_snapshot`／`preview_inspect`。
4. （美術項才有）identity／尺寸檢查——但美術全屬「人眼佇列」，故自動佇列不含。
5. 邊界主觀項：`/gpt` 第二意見（見 [[gpt-web-relay-skill]]）。

## 報告閘門（滿足任一才喚醒你）

1. **觸及「需你的眼睛」項目**——美術／角色語氣／主觀取捨。
2. **BLOCKED**——缺素材／模型／資訊，或需你拍板。
3. **無法自動修的退步**——validator 紅、runtime 壞、QA 修不回。
4. **里程碑摘要**——自動佇列清空，或累積一批做完 → 一份彙整，不逐項打擾。

非觸發（只記日誌）：validator＋runtime＋檔案檢查能客觀確認的例行打磨。

## 停止條件

碰到人眼項目／BLOCKED／不可回復的 QA 紅／佇列清空／迭代上限。

## 機制（怎麼跑）

- **有限批次**（目前自動佇列 4 項）→ 就地一回合連跑，逐項自動 QA，最後一份彙整。不需排程。
- **常駐推天花板** → `/loop` 動態自排程：每次醒來做下一項、自動 QA，只在閘門觸發時 ping 你。
